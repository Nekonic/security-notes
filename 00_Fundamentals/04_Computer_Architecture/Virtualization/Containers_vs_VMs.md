# 컨테이너 vs 가상 머신

둘 다 "격리된 실행 환경"을 주지만, **격리의 깊이가 근본적으로 다르다.** 보안 경계를 이해하려면 이 차이가 핵심이다.

---

## 1. 구조 비교

```
[가상 머신]                      [컨테이너]
┌──────┬──────┐                ┌──────┬──────┐
│ App  │ App  │                │ App  │ App  │
│ Bins │ Bins │                │ Bins │ Bins │
│ 게스트│ 게스트│                ├──────┴──────┤
│  OS  │  OS  │  ← 각자 커널!   │ 컨테이너 런타임│
├──────┴──────┤                ├─────────────┤
│ 하이퍼바이저  │                │  호스트 커널   │ ← 커널 공유!
├─────────────┤                ├─────────────┤
│   하드웨어    │                │   하드웨어    │
└─────────────┘                └─────────────┘
```

| | VM | 컨테이너 |
|---|----|----|
| 커널 | **VM마다 독립** 게스트 커널 | **호스트 커널 공유** |
| 격리 수단 | 하이퍼바이저 (하드웨어 보조) | namespace + cgroup (커널 기능) |
| 부팅/크기 | 초~분, GB | 밀리초, MB |
| 격리 강도 | **강함** (커널 분리) | 상대적으로 약함 (커널 공유) |
| 오버헤드 | 큼 | 작음 |

---

## 2. 컨테이너 격리의 정체 — 커널 기능일 뿐

컨테이너는 "가벼운 VM"이 아니다. **호스트 커널의 격리 기능으로 분리된 그냥 프로세스**다.

- **namespace**: 무엇을 *볼 수 있나* 를 분리 (PID, network, mount, user, UTS, IPC)
  → 컨테이너는 자기 프로세스/네트워크/파일시스템만 봄
- **cgroup**: 무엇을 *얼마나 쓰나* 를 제한 (CPU, 메모리, I/O)
- **capabilities / seccomp / LSM**: 무엇을 *할 수 있나* 를 축소 (위험한 시스템콜·권한 차단)

→ 이 모든 게 **하나의 공유 커널** 위에서 동작. 커널에 구멍이 나면 격리가 통째로 무너진다.

---

## 3. 보안 함의 — 공격면 차이

| | VM | 컨테이너 |
|---|----|----|
| 탈출 시 필요한 것 | 하이퍼바이저(QEMU/KVM) 취약점 | **호스트 커널** 취약점 1개 |
| 공격면 크기 | 작음 (좁은 하이퍼바이저 인터페이스) | **큼** (전체 syscall 인터페이스) |
| 탈출 결과 | 호스트·이웃 VM | 호스트·이웃 컨테이너 |

핵심: **커널 syscall 인터페이스는 하이퍼바이저 인터페이스보다 훨씬 넓다.** 그래서 컨테이너 탈출이 상대적으로 쉽다. 멀티테넌트(신뢰 안 되는 사용자 혼재) 환경에서 컨테이너만으로는 격리가 부족하다고 보는 이유.

---

## 4. 그 사이를 메우는 기술

신뢰 경계가 필요한데 컨테이너 속도도 원할 때:

| 기술 | 방식 |
|------|------|
| **Kata Containers** | 컨테이너를 경량 VM 안에서 실행 (커널 분리 + 컨테이너 UX) |
| **gVisor** | 유저공간에 커널을 다시 구현해 syscall을 가로챔 (공격면 축소) |
| **Firecracker** | 마이크로VM (AWS Lambda 기반) — 최소 장치, 빠른 부팅 |

→ "VM의 격리 + 컨테이너의 민첩성"을 노린다.

---

## 5. 클라우드 관점

- IaaS(OpenStack Nova)는 **VM 기반** → 테넌트 간 커널 분리로 강한 격리
- 컨테이너 오케스트레이션(K8s)에서 멀티테넌트면 → Kata/gVisor 같은 **샌드박스 런타임** 고려
- 격리가 깨지는 구체적 경로 → [Isolation_Security](Isolation_Security.md)
