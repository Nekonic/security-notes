# 메모리 가상화

게스트 OS는 자기가 0번지부터 시작하는 연속된 물리 메모리를 가진다고 믿는다. 하지만 실제 물리 메모리(RAM)는 여러 VM이 나눠 쓴다. 주소를 한 겹 더 번역해야 한다.

---

## 1. 문제 — 주소가 3종류가 된다

일반 OS는 주소가 2단계다:
```
가상주소(VA) ──[페이지테이블]──▶ 물리주소(PA)
```

가상화하면 게스트의 "물리주소"가 진짜 물리주소가 아니다. 한 단계가 더 생긴다:
```
게스트 가상주소(GVA)
   │ 게스트 페이지테이블 (게스트 OS가 관리)
게스트 물리주소(GPA)   ← 게스트는 여기가 진짜 RAM인 줄 안다
   │ ??? (하이퍼바이저가 관리해야 함)
호스트 물리주소(HPA)   ← 진짜 RAM
```

→ GPA를 HPA로 바꾸는 **두 번째 번역**을 누가 어떻게 하느냐가 핵심.

---

## 2. 해결 방식

### (1) Shadow Page Table (소프트웨어, 구식)
하이퍼바이저가 GVA→HPA를 **직접** 매핑하는 "그림자 페이지테이블"을 따로 유지.
- 게스트가 페이지테이블을 고치면 하이퍼바이저가 가로채(트랩) shadow를 동기화
- 문제: 동기화 비용이 크고 VM Exit가 잦음 → **느림**

### (2) 2단계 페이징 (하드웨어, 현재 표준) ⭐
CPU가 두 번째 번역을 하드웨어로 처리:
- **Intel: EPT** (Extended Page Table)
- **AMD: NPT** (Nested Page Table) / RVI

```
GVA ─[게스트 PT]→ GPA ─[EPT/NPT]→ HPA
       (게스트가 관리)    (하이퍼바이저가 관리, CPU가 자동 워크)
```

- 게스트가 자기 페이지테이블을 자유롭게 고쳐도 트랩 불필요 → **빠름**
- 단점: 두 단계를 다 워크하므로 TLB 미스 비용이 큼 (그래서 거대페이지로 완화)

---

## 3. IOMMU — 장치의 메모리 접근 가상화

CPU만 메모리를 쓰는 게 아니다. DMA(직접 메모리 접근) 하는 장치도 있다. PCI passthrough로 VM에 장치를 직접 주면, 그 장치가 **아무 물리주소나 DMA로 건드릴 수 있어** 위험.

→ **IOMMU** (Intel **VT-d** / AMD-Vi)가 장치의 DMA 주소도 번역·제한한다.
- 장치가 보는 주소(IOVA) → 실제 물리주소로 번역
- VM에 할당된 영역 밖은 장치가 못 건드리게 차단 → **passthrough 보안의 핵심**

---

## 4. 메모리 절약 기법과 그 위험

| 기법 | 내용 | 보안 위험 |
|------|------|-----------|
| **오버커밋** | 실제 RAM보다 많이 VM에 약속 | 고갈 시 스왑·OOM → DoS |
| **KSM (페이지 공유)** | 동일 내용 페이지를 VM 간 공유 | 공유 여부 타이밍으로 **사이드채널**(다른 VM 데이터 추정) |
| **벌루닝** | 게스트에서 메모리 회수 | 게스트 협력 필요 |

> 보안 강한 환경에서는 KSM을 끄거나 테넌트 간 공유를 막는다.

---

## 5. 보안 관점

- **EPT/NPT 매핑 오류·버그** → VM이 다른 VM의 HPA에 접근하면 격리 붕괴
- **IOMMU 미사용 passthrough** → 장치 DMA로 호스트 메모리 임의 접근 (매우 위험)
- **KSM 사이드채널** → 메모리 중복제거 타이밍으로 이웃 VM 정보 추정
- **잔존 데이터**: VM 삭제 후 물리페이지를 0으로 안 지우면 다음 VM이 읽음 → [Nova Ephemeral_Data](../../../06_Infra_Pentesting/06_Cloud_Security/OpenStack/Nova_Compute/README.md)
